How does this work?
Workflow

Mailtrack Hunter scans your inbox periodically. It pre-filters commercial emails by header (free), extracts image URLs, discards known CDNs and click-tracking domains, then asks Claude to classify only unknown silent pixels — tiny invisible images that silently report email opens. Together with the classification, Claude also suggests the optimal declarativeNetRequest urlFilter value for each tracker (visible as a small blue hint below the domain). Candidates appear here for your review before anything touches the extension. Already-known domains (any status) are never sent to Claude again.

Candidate actions
  • Pending — found by Claude, waiting for your review
  • Approve — confirms the domain is a real tracker; included in exports
  • Reject — marks it as a false positive; excluded from exports, kept as history
  • Reset — moves an approved or rejected candidate back to Pending
  • × — permanently deletes the candidate from the database
Toolbar
  • Scan now — triggers an immediate scan without waiting for the next interval
  • Auto-classify known — automatically approves or rejects pending candidates whose parent domain (e.g. example.com) already has a decision under a different subdomain
  • Purge rejected — permanently deletes all rejected candidates to keep the table clean
  • Select — enables multi-select mode with checkboxes for bulk approve / reject / delete / reset proposed; click Done to exit; the Reset proposed button appears automatically when at least one selected candidate is Proposed
Export & propose
  • Export full JSON — all approved candidates with full metadata (domain, source email, Claude's reasoning, timestamp); useful for auditing or feeding back to Claude
  • Export rules.json — approved domains as declarativeNetRequest rules, ready to merge into Block Mailtrack's rules/rules.json; each rule's urlFilter uses Claude's suggestion when available, falling back to a local heuristic for older candidates
  • Propose to Block Mailtrack (N) — opens a pre-filled GitHub Issue with as many approved candidates as fit within GitHub's URL limit (batch), skipping those already covered by Block Mailtrack's live rules (fetched from GitHub, refreshed every hour); candidates are marked as Proposed only after you confirm by entering the issue number; if more remain, the button stays active and shows the updated count — keep clicking until it reaches zero; click × next to the badge to reset individually, or use Select + Reset proposed in the bulk bar for multiple candidates at once
Maintenance API
  • GET /api/clean — removes pending candidates excluded by current filtering rules (ESP click-tracking domains, non-image extensions); call after updating scanner logic
  • GET /api/auto-classify-by-parent — same as the toolbar button, callable directly
  • POST /api/candidates/reset-proposed — resets Proposed status on a list of candidate IDs, making them available for re-proposing
